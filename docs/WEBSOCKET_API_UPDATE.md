# Техническое Задание: API Статуса Собеседника и Устойчивости Чата (WebSocket)

Этот документ описывает изменения в API, направленные на повышение устойчивости чата к кратковременным разрывам соединения. Основная цель — предотвратить немедленное завершение чата при дисконнекте одного из пользователей.

## 1. Обзор изменений

-   **НОВОЕ СОБЫТИЕ**: Добавлено событие `chat:partner_status`, которое информирует клиента об изменении статуса подключения его собеседника (`online` или `offline`).
-   **ИЗМЕНЕНИЕ ЛОГИКИ**: Событие `chat:ended` теперь отправляется не сразу, а только после 30-секундного "льготного периода", если пользователь не смог переподключиться.

## 2. Спецификация событий

### 2.1. Новое событие: `chat:partner_status`

-   **Направление**: Сервер → Клиент
-   **Назначение**: Уведомляет клиента о том, что его собеседник временно потерял соединение или восстановил его. Это позволяет фронтенду отобразить соответствующий статус (например, "Переподключение...") вместо того, чтобы сразу закрывать чат.

#### **Payload (Тип данных)**

```typescript
{
  chatId: string;
  userId: string;
  status: 'online' | 'offline';
}
```

-   `chatId`: Уникальный идентификатор чата, в котором произошло событие.
-   `userId`: Уникальный идентификатор пользователя, чей статус изменился.
-   `status`:
    -   `'offline'`: Собеседник отключился. В этот момент фронтенд должен показать индикатор переподключения.
    -   `'online'`: Собеседник успешно переподключился в течение льготного периода. Индикатор переподключения следует убрать.

---

### 2.2. Изменение поведения: `chat:ended`

-   **Направление**: Сервер → Клиент
-   **Назначение**: Окончательно завершает чат.

#### **Измененная логика сервера**

Событие `chat:ended` теперь отправляется **только в одном из двух случаев**:

1.  Один из пользователей штатно нажал кнопку "Завершить чат".
2.  Один из пользователей отключился (`offline`), и **не смог восстановить соединение в течение 30 секунд**.

Это значит, что фронтенд больше не должен получать `chat:ended` сразу после обрыва связи у собеседника.

## 3. Пример последовательности действий (сценарий)

1.  **User A** и **User B** находятся в активном чате.
2.  У **User B** пропадает интернет. Он отключается от сервера.
3.  Сервер немедленно отправляет **User A** событие:
    ```json
    {
      "type": "chat:partner_status",
      "payload": {
        "chatId": "чат_123",
        "userId": "id_юзера_B",
        "status": "offline"
      }
    }
    ```
4.  **User A** на своем клиенте видит сообщение "User B переподключается...". Чат при этом **не закрывается**.
5.  **Сценарий 1: Успешное переподключение (в течение 30 сек)**
    -   **User B** восстанавливает интернет, его клиент автоматически переподключается к серверу.
    -   Сервер отменяет таймер завершения чата и отправляет **User A** событие:
        ```json
        {
          "type": "chat:partner_status",
          "payload": {
            "chatId": "чат_123",
            "userId": "id_юзера_B",
            "status": "online"
          }
        }
        ```
    -   У **User A** исчезает плашка "Переподключение...". Чат продолжается как обычно.
6.  **Сценарий 2: Таймаут (30 сек прошли)**
    -   **User B** не смог восстановить соединение.
    -   Сервер по истечении 30-секундного таймера отправляет **User A** событие `chat:ended`:
        ```json
        {
          "type": "chat:ended",
          "payload": {
            "chatId": "чат_123",
            "endedBy": "id_юзера_B",
            "reason": "partner_disconnected"
          }
        }
        ```
    -   Клиент **User A** получает это событие и корректно завершает чат, показывая соответствующее сообщение.

## 4. Ответы на сообщения (Reply)

### 4.1. Обзор функциональности

Добавлена поддержка ответов на сообщения. Пользователи теперь могут отвечать на конкретные сообщения в чате.

### 4.2. Изменения в событии `chat:message`

#### **Отправка сообщения (Клиент → Сервер)**

```typescript
{
  chatId: string;
  content: string;
  replyTo?: string;  // ID сообщения, на которое отвечаем (опционально)
}
```

-   `chatId`: Уникальный идентификатор чата
-   `content`: Текст сообщения
-   `replyTo`: (опционально) ID сообщения, на которое отвечаем

#### **Получение сообщения (Сервер → Клиент)**

Структура объекта `message` в событии `chat:message` расширена:

```typescript
{
  chatId: string;
  message: {
    _id: string;
    chatId: string;
    sender: {
      _id: string;
      telegramId: string;
      username?: string;
      firstName?: string;
      lastName?: string;
      photos?: string[];
    };
    content: string;
    timestamp: Date;
    isRead: boolean;
    readBy: string[];
    replyTo?: {  // Присутствует только если это ответ на сообщение
      _id: string;
      content: string;
      timestamp: Date;
      sender: {
        _id: string;
        telegramId: string;
        username?: string;
        firstName?: string;
        lastName?: string;
      };
    };
  };
}
```

### 4.3. Пример использования

#### **Отправка обычного сообщения**

```json
{
  "type": "chat:message",
  "payload": {
    "chatId": "чат_123",
    "content": "Привет! Как дела?"
  }
}
```

#### **Отправка ответа на сообщение**

```json
{
  "type": "chat:message",
  "payload": {
    "chatId": "чат_123",
    "content": "Отлично! А у тебя как?",
    "replyTo": "сообщение_456"
  }
}
```

#### **Получение ответа на сообщение**

```json
{
  "type": "chat:message",
  "payload": {
    "chatId": "чат_123",
    "message": {
      "_id": "сообщение_789",
      "chatId": "чат_123",
      "sender": {
        "_id": "пользователь_2",
        "telegramId": "987654321",
        "firstName": "Алиса"
      },
      "content": "Отлично! А у тебя как?",
      "timestamp": "2024-01-15T12:35:00Z",
      "isRead": false,
      "readBy": [],
      "replyTo": {
        "_id": "сообщение_456",
        "content": "Привет! Как дела?",
        "timestamp": "2024-01-15T12:34:00Z",
        "sender": {
          "_id": "пользователь_1",
          "telegramId": "123456789",
          "firstName": "Боб"
        }
      }
    }
  }
}
```

### 4.4. Валидация

Сервер выполняет следующие проверки при получении сообщения с `replyTo`:

1. **Существование сообщения**: Проверяется, что сообщение с указанным ID существует
2. **Принадлежность к чату**: Проверяется, что оригинальное сообщение принадлежит тому же чату
3. **Доступность**: Пользователь должен быть участником чата

### 4.5. Ошибки

При невалидном `replyTo` сервер возвращает ошибку:

```json
{
  "type": "error",
  "payload": {
    "message": "Reply message not found"
  }
}
```

или

```json
{
  "type": "error",
  "payload": {
    "message": "Reply message is not from the same chat"
  }
}
```
